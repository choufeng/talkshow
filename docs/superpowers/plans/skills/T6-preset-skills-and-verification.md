# T6: 预置 Skill 与端到端验证

## 所属项目
[Skills 文本处理系统](../../specs/2026-03-30-skills-system-design.md)

## 依赖
- T3: 录音流程集成（后端管线就绪）
- T5: 导航集成（前端 UI 就绪）

## 目标
确保 4 个预置 Skill 的 prompt 内容经过精心编写和验证，并对整个 Skills 流程进行端到端验证。

## 任务详情

### 1. 预置 Skill 内容编写

为 4 个预置 Skill 编写经过测试的 prompt 内容：

#### 语气词剔除
```
去除中文口语中常见的无意义语气词和填充词，包括但不限于：
"嗯"、"啊"、"额"、"呃"、"那个"、"就是"、"然后"、"对吧"、"的话"、"怎么说呢"。
注意保留有实际语义的词语，例如"然后"在表示时间顺序时应保留。不要改变原文的语义和语气。
```

#### 错别字修正
```
识别并修正文本中的错别字、同音错误和常见输入法导致的文字错误。
只修正明确的错误，不要对有歧义的内容做主观改动。
常见的同音错误示例："的/地/得"、"做/作"、"在/再"、"已/以"、"即/既"。
```

#### 口语润色
```
保持口语化的表达风格，但使语句更流畅自然。
具体做法：去除重复表达、调整语序使其更通顺、适当添加标点使句子结构更清晰。
不要改变原文的口语化特征，不要转换为书面语。
```

#### 书面格式化
```
将口语化的表达转换为规范的书面表达，适合邮件、文档、报告等正式场景。
具体做法：
- 将口语化的词汇替换为正式表达
- 调整句子结构使其符合书面语法
- 适当分段和添加标点
- 保持原文的完整语义，不添加或删除信息
```

### 2. 端到端验证场景

#### 场景 A：基础流程
1. 配置 Skills Provider 和 Model
2. 启用"语气词剔除"和"错别字修正"
3. 录音说一段包含语气词和错别字的话
4. 验证：输出文字中无语气词，错别字已修正

#### 场景 B：应用感知
1. 启用所有 Skill（包括"口语润色"和"书面格式化"）
2. 在微信中录音 → 验证输出偏口语化
3. 在邮件/编辑器中录音 → 验证输出偏书面化

#### 场景 C：降级
1. 配置一个无效的 Skills Provider
2. 录音 → 验证输出原始转写文字，不报错
3. 检查日志中有 Skills 失败记录

#### 场景 D：全局关闭
1. 关闭 Skills 全局开关
2. 录音 → 验证行为与旧版完全一致

#### 场景 E：自定义 Skill
1. 在 UI 中添加一个自定义 Skill（如"翻译为英文"）
2. 启用它并禁用其他 Skill
3. 录音说中文 → 验证输出英文翻译

#### 场景 F：配置持久化
1. 添加/编辑/删除 Skill
2. 重启应用
3. 验证所有配置保持不变

### 3. 性能验证

- Skills 处理耗时记录在日志中
- 确认从转写完成到粘贴的总延迟不超过 15 秒（含 LLM 调用）
- 超时机制正常工作

## 验收标准

- [ ] 4 个预置 Skill 的 prompt 内容完整且有效
- [ ] 场景 A-E 全部验证通过
- [ ] 配置持久化正常（场景 F）
- [ ] 性能在可接受范围内
- [ ] 降级场景不阻塞主流程
- [ ] 日志记录完整
